JavaScript Coding Standards for 152-119

  • Here are the coding standards for this course, now enhanced to include loops and functions. Your projects and labs will be graded partially on conformance to these rules.
  • One of the most important aspects of modern programming is readability. To be able to read code written by other people and understand it when performing subsequent maintenance to it, has become much more important than having condensed, "clever" code. Modern programming languages can optimize your code better than you can. So, the number one criteria for the way we write is for other human beings to read it. These standards are about readability, thus enhancing the maintainablity of your code when making future changes to it.

1. Indentation

  • The unit of indentation of JavaScript source code will be four (4) spaces. This can be done by simply pressing the space bar four times. The course editor (jEdit) has been configured so that when you press the tab key it will insert 4 spaces instead of a tab. Keep this setting. However, if you use another text editor make sure you are not inserting tab characters into your code. And note: If for some reason the jedit tab key is not causing proper indentation, then don't use it.

  • If a line of code is too long to fit on one line of the text editor (see Line Length standard below), the continuation of the line must be indented in by eight (8) spaces to visually emphasize that this line is a continuation.

2. Line Length

  • Avoid making any line of JavaScript code more than 80 characters. This improves readability. Sometimes this can't be avoided but whenever possible it is required to have 80 or fewer characters in a line of code. 4. Constant Declarations

  • Constants must be declared before variables. If the name of the constant is not completely self-explanatory, use a comment on the right side of that line.
  • Other than the number 1, numeric literals may not be used in any processing or calculation. Instead, place the numbers in constants, and use the name of the constant to reveal the signficance of the number within the context of the application.
  • Constants use the var statement like variables.
  • Constants names will consist of uppercase letters only, with words separated by an underscore (_).
  • Constants are always assigned their values when they declared.
  • Each constant must be defined on a separate line.

5. Variable Declarations

Good variable naming and usage is very important when learning to program computers.

  1. Variable names must follow the naming conventions outlined in Part 8 below.
  2. All variables must be declared before they are used. Many programming languages require this as part of their syntax. However, JavaScript does not require this so you need do this intentionally.
  3. var statements must be at the top of your JavaScript code and functions, right after the constant declarations.
  4. Every variable must be declared on its own line. That is, one var statement per line, and only one variable declared on each var statement.

7. Function Declarations

Here are the coding standards for functions. (See also 8. below)

  • All functions must be declared in a javaScript library file (i.e. a file with an extension of .js). No functions are allowed in the HTML document.
  • All new functions created by students must have names that begin with a verb. These names must be in camel case format, and contain no abbreviated words.
  • All functions must be prceded by a comment block. The comment itself must begin with the phrase: "The purpose of this function is to " and then followed with a one or two sentence statement of what the function accomplishes.
  • Functions may not be set up within other functions.
  • All functions must be declared before they are used, i.e. when a function calls another function, the called function must be physically located above it in the js library file. (REMEMBER: To meet this standard, the "mainline" aka "main" aka "controller" function must be located at the very bottom of the js library file.)  This is an important performance standard. Your code will run without conformance with this standard so care must be taken to follow it.
  • There should be no space between the name of a function and the "(" (left parenthesis) of its parameter list. There should be one space between the ")" (right parenthesis) and the "{" (left curly brace) that begins the body of the function.  The statements comprising the body of the function itself are indented four spaces.  The "}" (right curly brace) is vertically aligned with the line containing the beginning of the declaration of the function (i.e. with the "f" in the word "function".)
  • return Statement (used in LU#5)
    • If used, a return statement with a value should not use ( ) (parentheses) around the value. The return value expression must start on the same line as the return keyword in order to avoid semicolon insertion.

8. Naming Conventions

How things are named in a computer program is very important. In the early days of programming names were very short and cryptic to save on RAM usage. You may have seen variables with names like "i", "fnme", and "cal_prc". These days our computers all have lots and lots of RAM and we can stop this old confusing habit. Here are the course rules for naming things:

  1. Variable and function names must be created with the 26 upper and lower case letters (A..Z, a..z) and the 10 digits (0..9).
  2. ( Remember: As stated above: The names of constants must be in ALL CAPS. )
  3. Do not use the underscore "_", the dollar sign "$", or slashes "/ \" in variable and function names. (Remember: constants, however, do use the underscore "_" to separate words.)
  4. Names of variables and functions must start with a lower case letter.
  5. As stated above, any new functions created by students must have names that begin with a verb.
  6. Names in general must NOT contain abbreviations. Instead, use complete words wherever possible, which is the vast majority of the time. Consider: What does "cal_prc" mean? Calculate Price? Calculate Percentage? Calender Perfection? Names need to be very clear, for others and for you. You won't remember your own cryptic names in the future.  (There is one exception when controlling a loop, where "i" could stand for the word "index" and is a fairly well-known convention. However, use of the full word "index" is preferable to the single-letter "i". Check with your instructor prior to using the letter "i" as a variable name, to determine if he/she is permitting this one abbreviation to be used.)
  7. Names, other than constants, that have more than one word should use the "camel case" convention. With this convention, the name starts with a lower case letter and then each new word in the name starts with an upper case letter. Examples:

    8. Remember: the names of functions that you create should always begin with a verb, as shown in the above examples. And remember that in this course, the mainline function does not begin with a verb, but rather its name reflects the name of javaScript file that contains it.

11. Whitespace

Whitespace can improve the readability of code. Here are the course standards for whitespace.

  • Use blank lines liberally--your code shouldn't be all bunched together.
  • Blank spaces should be use like this:
    • A keyword followed by a left parenthesis "(" should be separated by a space.
  • Most operators should have a space on either side.

  • Unary operators must NOT have spaces around them.
