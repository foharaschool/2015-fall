Learning Unit 2 JavaScript Coding Standards

(Initial) JavaScript Coding Standards for 152-119

Here are the initial coding standards for Learning Unit 1 and now Learning Unit 2 for this course. (The standards will later be expanded for subsequent Learning Units.)

Your projects and labs will be graded partially on conformance to these standards.

One of the most important aspects of modern programming is readability. To be able to read code written by other people and understand it has become much more important than having condensed, optimal code. Modern programming languages can optimize your code better than you can. So, the number one criteria for the way we write is for other human beings to read it. These standards are about readability, thus enhancing the maintainability your code when making future changes to it.

1. Indentation

The unit of indentation of JavaScript source code will be four (4) spaces. This can be done by simply pressing the space bar four times. The course editor (jEdit) has been configured so that when you press the tab key it will insert 4 spaces instead of a tab character. Keep this setting. However, if you use another text editor make sure you are not inserting tab characters into your code.

If a line of code is too long to fit on one line of the text editor (see the Line Length standard below), the continuation of the line must be indented in by eight (8) spaces to visually emphasize that this line is continued. If a concatenation operator is involved, it must be the first thing on the continued line (and not positioned as the last thing on the previous line).

2. Line Length

Avoid making any line of JavaScript code more than 80 characters. This improves readability. Sometimes this can't be avoided but when possible it is required to have 80 or fewer characters in a line of code. 3. Comments

Good documentation will be required in your labs and projects files. Here is a minimal list of what the instructor will be looking for. Quality counts!

  1. The student's name, section, and email at the top of lab and project files.
  2. A comment line at the start of each section within a program. These section comment lines must begin with a verb.

4. Variable Declarations

Good variable naming and usage is very important when learning to program computers. Here is a list of the rules for variables:

  1. Variable names must follow the naming conventions outlined in the Naming Conventions standard below.
  2. All variables must be declared before they are used. Many programming languages require this as part of their syntax. However, JavaScript does not require this so you need do this intentionally.
  3. All var statements must be at the top of your JavaScript code (and later,functions), before any other statements.
  4. Every variable must be declared on its own line. That is, one var statement per line, and only one variable declared on each var statement.
  5. No processing of any kind can be done on a var statement (e.g. no use of the prompt() method, nor any arithmetic operations). The most that can be done would be an assignment of a single number or text string.

5. Naming Conventions

How things are named in a computer program is very important. In the early days of programming names were very short and cryptic to save on RAM usage. You may have seen variables with names like "i", "fnme", and "cal_prc". These days our computers all have lots and lots of RAM and we can stop this old confusing habit. Here are the course rules for naming things:

  1. Names must bee created with the 26 upper and lower case letters (A..Z, a..z) and the 10 digits (0..9).
  2. Do not use the underscore "_", the dollar sign "$", or slashes "/ \" in variable names. (Note: Later on, we will have a different naming standard for constants, which will involve the use of the underscore "-".)
  3. Variables must start with a lower case letter.
  4. Abbreviations must be avoided--use complete words instead. What does "cal_prc" mean? Calculate Price? Calculate Percentage? California Practice? Calender Pricing? What does "temp" mean? Temporary? Temperature? Temporary Employee? Names need to be very clear, for others and for you. You won't remember your own cryptic names in the future when you return to make enhancements to your code.

8. Whitespace

Whitespace greatly improves the readability of code. Here are the course standards for whitespace.

  • Use blank lines liberally--your code shouldn't be all bunched together.
  • After each section of your program, leave a blank line.  (As mentioned earlier, each section of a program should begin with a comment line, with each such line beginning with a verb.)
  • Most operators must have a space on either side. Note that this includes the assign operator, all four arithmetic operators, the concatentation operator, the equality operators, and the relation operators. (It does NOT include the dot (.) operator, nor the plus plus (++) operator, nor the minus minus (--) operator, all of which must be written without blanks around them.)
  • A space should follow every comma ","

9.  Document.write() Only for Output of Variables and Concatenated Text

  • Use the document.write() method only to output the content of variables, with, if needed, explanatory text concatenated with the variables.
  • No mathematical operations may be done within the parenthese of a document.write(). Instead, do the math in a section of the progam above the document.write() methods, with the results stored into variables. Those variables are then used below within the document.write() method parentheses.
